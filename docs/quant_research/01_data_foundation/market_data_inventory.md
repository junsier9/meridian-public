### 1.4 `onchain_stablecoin_ethereum/` — Ethereum stablecoin daily plumbing aggregates (M3.2)

| Path                                      | Size  | Provider | Granularity | Symbols | Date range | Cols | Sync |
| ----------------------------------------- | ----: | -------- | ----------- | ------- | ---------- | ---- | ---- |
| `onchain_stablecoin_ethereum/daily_aggregates.csv` | \<2 MB (current) | Ethereum mainnet RPC / Alchemy Transfers fallback + local PIT label snapshot | 1d | 3 (`USDT`, `USDC`, `DAI`) | rolling trailing window, plus explicit historical backfill windows | 30 | `scripts/quant_research/sync_alchemy_stablecoin_ethereum.py` |

Schema:
```
date_utc, token_symbol, contract_address, decimals,
transfer_count, transfer_amount,
mint_count, mint_amount,
burn_count, burn_amount,
net_issuance_amount,
whale_transfer_count, whale_transfer_amount,
exchange_inflow_amount, exchange_outflow_amount, exchange_netflow_amount,
whale_to_exchange_amount, exchange_to_whale_amount,
issuer_to_exchange_amount,
bridge_inflow_amount, bridge_outflow_amount,
labeled_transfer_share_amount, unknown_transfer_share_amount,
unique_from_count, unique_to_count,
coverage_start_utc, coverage_end_utc, is_full_day, fetch_status, source
```

Interpretation:
- `transfer_amount` is token-unit volume; for stablecoins this is a close proxy for USD notional at parity.
- `mint_*` / `burn_*` come from zero-address ERC20 transfers and form the first M3.2 `MF-13 stablecoin_plumbing` supply-impulse track.
- `whale_*` uses the current script default threshold of `$1,000,000` equivalent per transfer.
- `exchange_*` / `whale_to_exchange_*` / `exchange_to_whale_*` are Phase 2 directional-flow fields and are only populated when the local address-label sidecar has a PIT-valid snapshot for the requested day.
- `labeled_transfer_share_amount` and `unknown_transfer_share_amount` let research audit how much daily notional is actually covered by the current label universe before any MF-13 / MF-14 admission decision.
- As of `2026-05-01`, the Phase 2 seed label set covers a narrow but non-zero exchange universe across `Coinbase`, `Binance`, `Kraken`, `ByBit`, `Bitfinex`, `OKX`, `KuCoin`, plus `Tether Treasury`.

## 7. Update Protocol — REQUIRED for all new market data

When adding any new market dataset (new venue, new symbol set, new granularity, new derived panel), the integrator **MUST** update this file in the same commit that introduces the data. Required additions:

1. **For new raw caches** (new external root or new provider): add row to §1 with path, size, provider, granularities, BTC (or anchor-symbol) date range, column count, and sync script. Update §6 sync script registry.
2. **For new committed artifacts** (new files under `artifacts/`): add row to §2 with path, provider, granularity, symbols, date range, column count, consumer.
3. **For new derived panels**: add row to §3 with builder module path, row count, subject count, date range, downstream consumer (which score / overlay / cycle).
4. **For new providers / API keys**: add row to §4 with env var name, usage, status.
5. **For new symbols added to existing top-30 universe**: update §5 coverage matrix row.
6. **For new sync scripts**: add row to §6.

After updating, also append an entry to `config/quant_research/threshold_provenance.md` referencing this inventory file for the new data's audit lineage.

### Common pitfalls to avoid

- **Don't put new data under `market_history/coinapi_ohlcv/`** if it's a non-Binance exchange — that overwrites the Binance catalog. Use a per-exchange root like `coinapi_ohlcv_<EXCHANGE>/` instead (see §1.2 pattern).
- **Don't forget `.gitignore` exception lists** — `artifacts/quant_research/*` is gitignored by default; explicit exceptions (like `artifacts/external_market_data/deribit_dvol/`) need a `!` rule. Most derived panels stay gitignored (regenerable); only commit when the data has external-source provenance not derivable from sync.
- **Don't commit raw OHLCV / derivatives caches** to git — they're typically 100+ MB and regenerable from the sync scripts. Keep `LOCALAPPDATA` as the canonical raw store.
- **Date ranges in this inventory are anchored to BTC or the alphabetically-first-symbol's coverage** — for per-asset coverage variance, see actual partition listings in the cache directory.
